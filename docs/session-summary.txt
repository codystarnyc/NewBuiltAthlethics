1. PRIMARY REQUEST AND INTENT
------------------------------
The primary intent is to develop an AI-Powered Fitness & Nutrition App, a mobile-first platform that tracks workouts, uses image recognition for logging food/body/receipts, generates AI-powered meal plans, and syncs with backend services.

The task progressed through these major phases:
- Mobile-First App Scaffold + Navigation + API Client
- ImageUploadAPI Rebuild + AI Layer

- Admin Dashboard Development (HTML/CSS/JS)
- Admin Dashboard Deployment to Vercel
- Database Migration from Supabase to AWS RDS PostgreSQL
- API Deployment to AWS Elastic Beanstalk

- Mobile App Core Screen Wiring (GymDiary, FoodDiary, Profile)

7. ERRORS AND FIXES
---------------------
- Supabase connection issues (Tenant not found, IPv6 unreachable) → Switched to AWS RDS
- AWS CLI not installed → brew install awscli
- IAM unauthorized for EC2/RDS → Attached AmazonRDSFullAccess, AmazonEC2FullAccess, AdministratorAccess
- RDS Free Tier restrictions → Set backup-retention-period 1, --no-storage-encrypted
- Invalid PostgreSQL version 16.4 → Used 16.6
- Stale Prisma client after RDS migration → Killed old server, regenerated Prisma client
- EB AccessDenied → Created required IAM roles and instance profiles
- EB npm install timeout on t3.micro → Upgraded to t3.small
- Mixed content (HTTPS dashboard → HTTP API) → Vercel rewrite proxy in vercel.json
- Vercel proxy returning HTML instead of JSON → Consolidated routes into rewrites
- Browser caching stale config → Hard refresh (Cmd+Shift+R)


8. WHAT'S NEXT (updated Apr 2, 2026)
-------------------------------------

Verified this session:
- Mobile `BuiltAthleticsMobile/src/api/endpoints.ts` still uses
  `process.env.EXPO_PUBLIC_API_URL ?? 'http://localhost:5000'` — production
  builds must set EXPO_PUBLIC_API_URL (e.g. `.env` locally, EAS secrets for
  store builds). iOS App Transport Security blocks plain HTTP to remote hosts
  unless you add exceptions; prefer HTTPS on the API before wide device testing.

High priority (order suggested):
1. HTTPS for the API — ACM + Application Load Balancer (or CloudFront in front
   of EB) so mobile and third-party clients can use a stable https:// URL.
2. Set mobile EXPO_PUBLIC_API_URL to that HTTPS API URL; confirm Android
   cleartext / iOS ATS as needed during transition.
3. Populate exercise library via admin dashboard (categories + exercises with
   images/videos) so Gym Diary and programs are usable with real data.
4. Real-device QA — auth, food diary sync, workout logging, meal plans, image
   upload/AI flows against the deployed API.

Medium priority:
5. Custom domains — api.* on EB/ALB, admin.* on Vercel; update admin dashboard
   proxy rewrites if the API hostname changes.
6. Edamam (or equivalent) API keys for barcode and nutrition lookups.
7. Seed or import food database (USDA / commercial dataset) for search quality.
8. Push notifications (FCM / APNs) for reminders and engagement.
9. Stripe (or chosen provider) for rewards marketplace checkout.

Before App Store / Play Store launch:
10. App icons, splash, and store-required screenshots/metadata.
11. EAS Build profiles (development / preview / production) with correct env
    vars and API URL per channel.
12. Rotate to a strong production JWT secret; ensure it matches all deployed
    API instances.
13. Security pass — tighten RDS security group (API subnets only), review rate
    limits, validate/sanitize inputs on all public routes.
14. Observability — CloudWatch alarms, error aggregation (e.g. Sentry), uptime
    checks on health endpoint.
15. Optional: API integration tests + CI (lint/build/test on PR) before
    scaling collaboration.
